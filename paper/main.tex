\begin{abstract}
Building useful knowledge bases from text has been a challenge for a long time. The standard process, known as the Knowledge Acquisition Bottleneck, requires a domain expert to read documents and write down entities and relations by hand. Large Language Models (LLMs) can now do most of this work in a single API call. This paper describes a web application that takes any text and returns an interactive knowledge graph. The system has four main design choices. First, it uses an open vocabulary for entity types with a controlled vocabulary of 23 relations. We started with 15 relations (the typical encyclopedic set) and added 8 more after we noticed that many biographical facts were collapsing to a generic \texttt{related\_to} edge. Second, the prompt tells the model to use \emph{event reification} for biographical events such as births and marriages, so that a person is never directly connected to a date. Third, the prompt asks the model to merge surface mentions of the same entity using an \texttt{aliases} field. Fourth, every edge carries a verbatim evidence sentence and a self-reported confidence in $[0,1]$. Additionally, The frontend itself also includes an \emph{importance-ranked complexity slider} that hides less central entities at lower levels, which is very useful for paper-ready summary views. The system is designed to work with four LLM providers (Anthropic Claude, OpenAI GPT, Google Gemini, and local Ollama models) behind one shared schema, and exports the graph as JSON, PNG, or RDF/Turtle.


\end{abstract}

\section{Introduction}\label{sec:intro}

Building queryable knowledge bases from text has historically been slow and expensive. Human experts need to read every document, decide which entities and relations are relevant, and write them down. Feigenbaum~\cite{feigenbaum1977art} called this the \emph{Knowledge Acquisition Bottleneck}, and the problem has not really disappeared.

LLMs change this in an special way. They can recognize entities and relations in any new passage without being trained on a specific datasets, and they can be asked to return structured output (JSON with a fixed schema) using tool-use APIs~\cite{anthropic2024toolUse} or strict JSON-Schema decoding~\cite{openai2024structured}. This paper describes a web application that uses this capability to do most of what a knowledge engineer used to do by hand.

The main contributions are:
\begin{enumerate}[leftmargin=1.8em]
  \item A hybrid extraction schema (Section~\ref{sec:design}) with open entity types and 23 controlled relations.
  \item A decision-rule system prompt (Section~\ref{sec:prompt}) that tells the model which relation to use for each common pattern, and that asks it to do event reification, alias merging, and generic/specific deduplication during extraction.
  \item A per-edge audit trail (Sections~\ref{sec:design},~\ref{sec:viz}): every edge has a verbatim evidence sentence and a confidence value in $[0,1]$.
  \item An interactive visualization (Section~\ref{sec:viz}) with degree-based node sizing, an ego-network highlight on hover, an importance-ranked complexity slider, and several filters that compose together.
  \item A provider-agnostic backend (Section~\ref{sec:architecture}) covering Anthropic, OpenAI, Gemini, and Ollama behind the same schema, with RDF/Turtle export.
\end{enumerate}

\section{Related Work}\label{sec:related}

Automated knowledge graph construction has gone through three main phases, and a few other research areas are also relevant for the choices we make below.

\paragraph{Rule-based and supervised pipelines.}
Early systems combined Named Entity Recognition with a separately trained Relation Extraction model. Mitchell et al.'s NELL system~\cite{mitchell2018neverending} showed that a semi-automated pipeline can keep building a knowledge base for years, but it still needed regular human curation, and it did not generalize well across domains. Adding a new relation meant collecting and labeling new training data.

\paragraph{Knowledge graph embeddings.}
A parallel line of work focused on learning vector representations for entities and relations. TransE~\cite{bordes2013translating} is the classic example: head + relation $\approx$ tail in the embedding space. Later models such as RotatE and ComplEx improved the geometry. These methods are mainly for reasoning over an existing graph (predicting missing links, finding inconsistencies). They are not directly useful for building a graph from raw text.

\paragraph{LLM-based extraction.}
Recent work uses LLMs as zero-shot or few-shot extractors. Pan et al.~\cite{pan2024unifying} survey this area and argue that LLMs and KGs should be combined: LLMs provide natural-language understanding, KGs provide structure and queryability. They specifically recommend controlled relations with open entity types, which is the choice we follow in Section~\ref{sec:design}. Microsoft's GraphRAG~\cite{edge2024graphrag} combines LLM extraction with graph-aware retrieval to improve question answering, and industry tools such as Neo4j's LLM Graph Builder~\cite{neo4j2024llmgraphbuilder} package the same idea for users. Our system fits in this third phase, but we focus on three things that other tools handle less explicitly: vocabulary coverage as an auditable property, n-ary reification done inside the prompt, and an importance-based summary view.

\paragraph{Ontology design and n-ary relations.}
Noy and McGuinness's \emph{Ontology Development 101}~\cite{noy2001ontology} explains the standard way to handle relations that have more than two arguments: introduce an extra node for the event itself, and attach the arguments to that node. A marriage, for example, involves two people, a date, and sometimes a place. In our system, the prompt asks the LLM to do this reification at extraction time rather than as a post-processing step.

\paragraph{Entity resolution.}
Ji et al.~\cite{ji2022survey} survey knowledge graph construction and completion. They list entity resolution (deciding when two surface forms refer to the same entity) as one of the recurring failure modes of automated pipelines. The Maria Sk\l odowska / Marie Curie case from their survey is exactly the problem we address with the \texttt{aliases} field (Section~\ref{sec:aliases}).

\paragraph{Knowledge graph summarization.}
Visualization research has noticed for a long time that even small KGs become hard to read after a few dozen nodes. Liu et al.~\cite{liu2016kgconstruction} review KG construction techniques and discuss summarization options based on structural features (degree, centrality) or semantic ones (query relevance, ontology depth). Our complexity slider (Section~\ref{sec:complexity}) is structural: it ranks entities by degree plus mention frequency, which is simple to compute and easy to explain.

\section{System Architecture}\label{sec:architecture}

The project is a small web application with two parts: a backend that exposes the extraction API, and a frontend that handles the input, the visualization, and the export. They communicate over JSON HTTP. Figure~\ref{fig:arch} shows the high-level data flow.

$\xrightarrow{\text{extract request}}$ &
\fbox{\parbox{3.2cm}{\centering Backend API\\\& provider router}} \\

& & \fbox{\parbox{3.6cm}{\centering LLM provider\\(Claude / GPT / Gemini / Ollama)}}

Only one LLM call is made per extraction; everything else (filtering, layout, complexity slider, export) happens in the browser.}
\label{fig:arch}

The backend integrates with four LLM providers: Anthropic Claude, OpenAI GPT, Google Gemini, and local models served by Ollama. Each provider has a different way of returning structured output, but the backend hides these differences behind a single common interface. The user can switch between providers from a dropdown menu in the UI without any change to the application. A small registry detects automatically which providers are available (based on which API keys are configured) and shows only those in the menu.

\section{Knowledge Model and Schema Design}\label{sec:design}

The most important design choices in this kind of system are the knowledge model (what counts as an entity, what counts as a relation, what extra fields each one carries) and the prompt that asks for it. 
\subsection{Hybrid Schema: Open Types, Controlled Relations}

A knowledge model can be \emph{closed}, where both entity types and relation labels are fixed in advance, or \emph{open}, where the extractor invents them. Closed models give clean, queryable graphs but only work in one domain. Open models handle any text but suffer from \emph{label drift}: the same relation appears as \textit{works\_at}, \textit{employed\_by}, and \textit{is\_employee\_of} in different runs, which breaks SPARQL queries and any union operation.

We use a \emph{hybrid}: open entity types, but controlled relations. The model picks the most natural type for each entity (Person, Element, Theory, Field of study, etc.), while relation labels come from a fixed list. This split is also what Pan et al.~\cite{pan2024unifying} recommend.

\subsection{Controlled Relation Vocabulary}\label{sec:vocab}

The vocabulary contains 23 relations chosen to cover the patterns that appear most often in biographical, scientific, and organizational text. They fall into three rough categories:

\begin{itemize}[leftmargin=1.5em]
  \item \textit{Biographical place relations.} A person is \emph{born in} a city, or \emph{lived in} a country. Using a single \texttt{located\_in} for both organizations and people would be a type error, so we keep them separated.
  \item \textit{Discovery vs.\ creation.} Natural elements, particles, and laws are \emph{discovered}, not \emph{created}. We provide \texttt{discovered\_by} alongside \texttt{created\_by} and \texttt{developed\_by} so that elements like polonium are not assigned a human-authorship relation.
  \item \textit{Shared honours.} When a Nobel Prize is shared between three people, the symmetric co-recipient relation needs to survive as more than a generic connection.
\end{itemize}

The full list is:

\begin{quote}\small
\texttt{works\_for}, \texttt{member\_of}, \texttt{founded\_by}, \texttt{located\_in}, \texttt{headquartered\_in}, \texttt{part\_of}, \texttt{created\_by}, \texttt{developed\_by}, \texttt{discovered\_by}, \texttt{causes}, \texttt{influences}, \texttt{occurred\_at}, \texttt{occurred\_on}, \texttt{has\_property}, \texttt{related\_to}, \texttt{opposes}, \texttt{born\_in}, \texttt{lived\_in}, \texttt{married\_to}, \texttt{studied\_at}, \texttt{awarded}, \texttt{awarded\_for}, \texttt{co\_recipient\_with}.
\end{quote}

A generic \texttt{related\_to} relation is included so the model can still emit a fact when none of the other 22 relations fits.

\subsection{Output Schema}

The extraction target is a JSON object with two arrays. Listing~\ref{lst:schema} shows the schema in abbreviated form.

\begin{lstlisting}[label=lst:schema, caption={Output schema (abbreviated). Each node carries its source sentences and any aliases. Each edge carries the verbatim evidence sentence and a confidence value in [0,1].}]
{
  "nodes": [{
    "id": "marie_curie",
    "label": "Marie Curie",
    "type": "Person",
    "description": "Polish-born French physicist and chemist...",
    "source_sentences": ["Marie Curie was a Polish-born physicist..."],
    "aliases": ["Maria Sklodowska"]
  }],
  "edges": [{
    "source": "polonium",
    "target": "marie_curie",
    "relation": "discovered_by",
    "evidence": "...they discovered the elements polonium and radium.",
    "confidence": 0.95
  }]
}
\end{lstlisting}

Three things to note. First, every single node has source sentences and every edge has an evidence sentence. This lets the user check the extraction, and it also makes the model less likely to invent facts: when you ask the model to cite a sentence, it tends to stay closer to the text. Second, the \texttt{id} field uses \texttt{lowercase\_with\_underscores}, which works directly as a URI fragment in the RDF/Turtle export (\texttt{kg:marie\_curie}). Third, \texttt{aliases} and \texttt{confidence} are required (not optional) in the schema, because OpenAI's \texttt{strict:true} mode does not allow optional properties; the model returns \texttt{[]} or a default of $0.7$ when there is nothing to say.

\section{The Prompt}\label{sec:prompt}

Beyond listing the relation vocabulary, the prompt does five things, each one designed to solve a specific extraction problem.

\subsection{Decision Rules for Each Relation}\label{sec:decision-rules}

To prevent over-use of the generic \texttt{related\_to} relation, the prompt includes short decision rules for the more easily confused relations. For example:

\begin{quote}\small\itshape
A \textsc{person} born in a place $\rightarrow$ \texttt{born\_in} (NEVER \texttt{located\_in}).\\
A \textsc{natural} phenomenon or element identified by a scientist $\rightarrow$ \texttt{discovered\_by}, not \texttt{created\_by}.\\
Two or more people sharing an award $\rightarrow$ \texttt{co\_recipient\_with} between each pair.
\end{quote}

\subsection{Event Reification}\label{sec:reification}

Biographical events such as births and marriages usually have more than two arguments (a person, a date, sometimes a place). Since each edge is binary, attaching them directly to one person produces type errors like \texttt{marie\_curie occurred\_on 1867}, which reads as ``Marie Curie happened on 1867''.

The standard fix from Noy and McGuinness~\cite{noy2001ontology} is to introduce an intermediate node for the event and attach the arguments to it (e.g.\  \texttt{birth\_marie\_curie occurred\_on 1867}, 
\texttt{birth\_marie\_curie occurred\_at warsaw}). The prompt asks the model to perform this reification at extraction time. The cost is more nodes; we offset this in the visualization by rendering event nodes smaller and hiding them first when the complexity slider is moved down (Section~\ref{sec:complexity}).

\subsection{Entity Resolution via Aliases}\label{sec:aliases}

When the same entity has multiple surface names (``Maria Sk\l odowska'' / ``Marie Curie''), a naive extractor produces disconnected nodes. The prompt instructs the model to emit a single node with the canonical label and list alternatives under an \texttt{aliases} field. In the side panel, aliases appear as small italic pills under the main label.

\subsection{Deduplicating Generic and Specific Mentions}

When a class and its instances both appear (``the Nobel Prize'' alongside ``the Nobel Prize in Physics''), the prompt asks the model to prefer the specific instances and link them with \texttt{part\_of} to the class node only if the class is worth keeping on its own.

\subsection{Confidence}\label{sec:confidence}

Each edge carries a self-reported \texttt{confidence} value in $[0,1]$: $1.0$ for relations stated with an unambiguous verb, $0.7$ for relations clearly implied, $0.4$ for relations inferred from context. Edges with confidence below $0.6$ render as dashed lines.

\section{Interactive Visualization}\label{sec:viz}

The frontend is a single page that puts the text input and provider menus on the left and the graph view on the right. The graph is drawn with Cytoscape.js, with a toolbar above it and a side panel that opens when a node is clicked.

\subsection{Visual Encoding}

Node color encodes the entity type (Person, Organization, Location, Concept, Event, plus a default). Node size is proportional to degree, so the most central entities visually stand out. Edge style encodes confidence: solid for high-confidence edges and dashed for low confidence. Event-reification nodes are rendered smaller and semi-transparent, so they fade behind the real entities. Edge labels are hidden by default and revealed only on hover, otherwise the graph becomes unreadable when there are many edges.

\subsection{Layouts}

Five layouts are available: force-directed, concentric (rings by degree), hierarchy, circle, and grid. The default is chosen automatically: concentric if one node has at least 40\% of the possible degree share (a typical hub), otherwise force-directed.

\subsection{Filters}

The toolbar has four filters that can be combined freely:

\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Relation chips:} hide all edges of a chosen relation.
  \item \textbf{Per-entity hide:} a ``hide'' button in the side panel removes a specific entity; hidden entities reappear from the toolbar with one click.
  \item \textbf{Dates toggle:} hides nodes whose label looks like a year, useful for clean screenshots.
  \item \textbf{Complexity slider:} described next.
\end{itemize}

\subsection{Importance-Ranked Complexity Slider}\label{sec:complexity}

The slider trims the graph down to its most important entities. Each entity gets a score equal to its degree plus twice the number of source sentences in which it is mentioned:
\[
\mathrm{score}(n) = \deg(n) + 2 \cdot |\mathrm{source\_sentences}(n)|.
\]
The mention term makes real entities score higher than event-reification nodes (which usually have no source sentences), so reified events are hidden first when the slider is moved down. The number of levels scales with graph size (3 for tiny graphs, up to 7 for large ones), and at each level only the top-ranked entities and the edges between them are shown. The slider composes with the manual hide list via a simple set union, so a peripheral entity hidden at maximum complexity stays hidden when the slider is dropped.

\subsection{Ego Network and the Side Panel}

Hovering a node fades the rest of the graph and highlights its closed neighborhood. Clicking a node opens the side panel, which shows the entity's label, any aliases, type, description, source sentences, and the lists of incoming and outgoing relations. Each relation row has a confidence badge.

\subsection{Export}

Three buttons in the graph header: JSON (the normalized graph), RDF/Turtle (with reified evidence and confidence), and PNG (a 2$\times$-resolution snapshot, used for the figures in this paper).

\section{Qualitative Evaluation}\label{sec:evaluation}

We evaluated the system on three passages of different character: a simple organizational sentence, a longer encyclopedic paragraph on climate science, and a biographical paragraph that uses an alias. The outputs were inspected manually for entity coverage, relation correctness, and alias resolution. Table~\ref{tab:results} summarizes the size of each extracted graph.

\subsection{Sample 1: Microsoft}

\emph{``Microsoft is headquartered in Redmond, Washington. Bill Gates founded the company, which develops Windows and Office.''}

The extraction is clean: \texttt{microsoft headquartered\_in redmond}, \texttt{redmond located\_in washington}, \texttt{microsoft founded\_by bill\_gates}, \texttt{windows developed\_by microsoft}, \texttt{office developed\_by microsoft}. Microsoft is connected to four of the five other entities (hub ratio $= 0.8$), so the page automatically picks the concentric layout. The result confirms that the controlled vocabulary covers simple organizational text.

\subsection{Sample 2: Climate Science}

\emph{``The greenhouse effect is a natural process in which gases in Earth's atmosphere trap heat from the Sun. Carbon dioxide, methane, and water vapor are the main greenhouse gases. Since the Industrial Revolution, human activities such as burning fossil fuels and deforestation have increased atmospheric carbon dioxide concentrations significantly. This enhanced greenhouse effect causes global warming, which leads to rising sea levels, more frequent extreme weather, and ocean acidification. The Intergovernmental Panel on Climate Change was established in 1988 by the United Nations to assess scientific evidence on climate change.''}

The second passage is a short paragraph on climate science taken from a popular-science introduction. The extraction with Gemini 2.5 Flash returns 25 entities and 20 relations in about 45 seconds. The type mix is varied (2 organizations, 1 location, 7 concepts, 2 events, 13 other), which is what the open-type design is built for: the model is not forced to pick from a closed list and uses a generic \emph{Other} type when nothing precise fits.

The relations show a clear chain: \emph{enhanced greenhouse effect} \texttt{causes} \emph{global warming}, which in turn \texttt{causes} \emph{rising sea levels}, \emph{extreme weather}, and \emph{ocean acidification}. Compositional facts are captured with \texttt{part\_of} (\emph{methane}, \emph{carbon dioxide}, and \emph{water vapor} as part of \emph{greenhouse gases}). Two events are reified: the \emph{establishment of the IPCC} node carries an \texttt{occurred\_on} edge to the date \emph{1988} and a \texttt{founded\_by} edge to the \emph{United Nations}, which is exactly the n-ary pattern the prompt is designed to produce. A few \texttt{related\_to} edges remain (for example, from \emph{scientific evidence} to \emph{climate change}), since the controlled vocabulary does not cover every relation in this domain.

This passage is also a good test of the importance-ranked complexity slider. Since $\lceil 25/5 \rceil = 5$, five levels are available; at level 1 the user sees only the most central nodes such as \emph{greenhouse effect}, \emph{global warming}, and \emph{greenhouse gases}, and at level 5 the full graph is shown for detailed inspection.

\subsection{Sample 3: Mark Twain}

\emph{``Mark Twain, born Samuel Clemens in Missouri, wrote The Adventures of Tom Sawyer and Adventures of Huckleberry Finn. He married Olivia Langdon in 1870, and the couple settled in Connecticut. Twain received an honorary doctorate from Oxford University in 1907.''}

The model emits a single \texttt{mark\_twain} node with \texttt{Samuel Clemens} stored as an alias, which is the expected outcome of the alias-resolution rule in the prompt. The biographical relations fire as designed: \texttt{married\_to} (Olivia Langdon, symmetric in both directions), \texttt{created\_by} for the two novels, and \texttt{awarded} for the honorary doctorate. Birth, marriage, settlement, and award are reified as their own event nodes, each carrying \texttt{occurred\_on} (date) and \texttt{occurred\_at} (place) attributes, which avoids the type errors of attaching dates directly to Twain. One \texttt{related\_to} edge connects the award event to \texttt{Oxford University}, since the vocabulary has no precise \texttt{awarded\_by} relation.

\section{Conclusion}\label{sec:conclusion}

We have built an LLM-powered knowledge graph extractor that takes any text and produces an interactive graph in seconds. The extraction layer uses a hybrid model (open entity types, 23 controlled relations), in-prompt event reification, alias-based entity merging, generic/specific deduplication, and self-reported confidence. The interaction layer uses degree-based node sizing, an ego-network highlight on hover, five layouts with a smart default, four orthogonal filters, an importance-ranked complexity slider, and JSON / RDF / PNG export.

The main lesson from the project is that classical knowledge-engineering problems do not disappear when you use a modern solutions like using LLMs. The vocabulary still drifts, n-ary relations still need reification, the same entity still appears under multiple names, and a graph that does not show its uncertainty is still hard to trust. What changes is who does the work: a single LLM call now does most of what a human knowledge engineer used to do by hand, and the engineer's job becomes designing the rules (the vocabulary, the prompt, the audit) that make the extraction trustworthy.
